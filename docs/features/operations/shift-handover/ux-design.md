## 1. 用户角色分析

### 1.1 站长 (station_master)

**基本信息：**
- 典型画像：30-45岁，持危险化学品经营许可证负责人资格证
- 工作环境：站点现场为主，办公室与巡视交替
- 使用频率：每日中频（班次交接时集中使用）
- 技术熟练度：中等，熟悉站点业务流程

**核心诉求：**
- 主要目标：监督交接班流程，审核异常情况，确保站点平稳运营
- 痛点：
  - 需要随时掌握各班次的经营数据
  - 异常交接需要快速响应和处理
  - 现金解缴审核需要及时完成
- 期望：一键查看所有班次状态，异常情况自动提醒，移动端支持

**场景约束：**
- 时间压力：中等，班次交接时段较忙
- 注意力分散：高，经常被现场事务打断
- 专业知识：熟悉站点所有业务流程

---

### 1.2 班组长 (shift_leader)

**基本信息：**
- 典型画像：28-40岁，具备加气站操作资质，责任心强
- 工作环境：站点现场，主要在收银台和监控室
- 使用频率：高频，每个班次开始和结束时必用
- 技术熟练度：中等，对常用功能熟练

**核心诉求：**
- 主要目标：高效完成交接班流程，确保数据准确交接
- 痛点：
  - 交接班流程步骤多，担心遗漏
  - 现金盘点时需要快速核对
  - 交接时发现问题不知如何记录
- 期望：步骤式引导交接流程，自动计算现金差额，便捷的问题记录

**场景约束：**
- 时间压力：高，班次交接时间有限
- 注意力分散：高，可能同时处理客户或设备问题
- 专业知识：熟悉本班次业务操作

---

### 1.3 收银员 (cashier)

**基本信息：**
- 典型画像：22-35岁，主要负责收银和客户服务
- 工作环境：收银台
- 使用频率：工作期间持续使用
- 技术熟练度：中等偏高，日常操作熟练

**核心诉求：**
- 主要目标：实时了解本班次销售情况
- 痛点：
  - 无法随时查看当前班次数据
  - 担心现金收取时出错
- 期望：简洁的班次汇总显示，清晰的金额展示

**场景约束：**
- 时间压力：高，需要同时服务客户
- 注意力分散：高，需要多任务处理
- 专业知识：理解基本收银流程

---

### 1.4 财务人员 (finance)

**基本信息：**
- 典型画像：28-45岁，财务/会计专业背景
- 工作环境：办公室
- 使用频率：每日1-2次，月末集中
- 技术熟练度：中等，擅长数据分析

**核心诉求：**
- 主要目标：审核现金解缴，核对财务数据
- 痛点：
  - 需要逐个查看待审核的解缴记录
  - 异常解缴需要详细了解原因
- 期望：待办清单一目了然，详细的审计信息

**场景约束：**
- 时间压力：月末较忙
- 注意力分散：低，专注数据核对
- 专业知识：理解财务指标

## 2. 核心任务流程

### 任务 1：查看当前班次汇总 (US-001)

**用户目标：** 实时了解本班次的经营数据
**使用频率：** 每班次多次
**时间压力：** 中等

**优化后流程：**

```
[进入首页/汇总页] → [查看核心指标卡片] → [查看详细分类] → [手动刷新(可选)]
        ↓                   ↓                   ↓               ↓
   自动显示当前班次    金额/单数/量一眼可见   支付方式/燃料分类    20秒自动刷新
```

**具体设计决策：**

| 步骤 | 用户行为 | 设计决策 | 依据 |
|------|---------|---------|------|
| 1 | 进入页面 | 自动识别当前班次，无需手动选择 | 减少操作步骤 |
| 2 | 查看核心指标 | 大字号显示营业额、笔数、充装量 | 最重要的信息最突出 |
| 3 | 查看分类明细 | 卡片式展示支付方式和燃料类型汇总 | 分类清晰，便于核对 |
| 4 | 数据刷新 | 自动20秒刷新 + 手动刷新按钮 | 平衡实时性和性能 |

### 任务 2：执行交接班 (US-004 ~ US-007)

**用户目标：** 完成班次交接，移交工作给下一班
**使用频率：** 每班次1次
**时间压力：** 高

**优化后流程：**

```
[点击"交接班"] → [预检确认] → [现金解缴] → [填写备注] → [选择接班人] → [确认交接]
       ↓              ↓            ↓           ↓             ↓            ↓
  显眼的入口     自动检查待办    自动计算差额   提供常用模板   下拉选择     二次确认
```

**具体设计决策：**

| 步骤 | 用户行为 | 设计决策 | 依据 |
|------|---------|---------|------|
| 1 | 发起交接班 | 汇总页固定位置的"交接班"按钮 | 高频操作，位置固定易形成肌肉记忆 |
| 2 | 查看预检 | 步骤向导式界面，清晰显示检查项及状态 | 防止遗漏，给用户安全感 |
| 3 | 现金解缴 | 系统自动计算应收，用户输入实收，自动计算差额 | 减少人工计算，降低错误 |
| 4 | 填写备注 | 提供常用模板（设备正常、库存充足等） | 加速输入，标准化描述 |
| 5 | 选择接班人 | 下拉列表显示下一班次可选员工 | 简化选择，防止输入错误 |
| 6 | 确认交接 | 显示汇总信息，二次确认弹窗 | 重要操作双重确认 |

### 任务 3：现金解缴 (US-010 ~ US-012)

**用户目标：** 完成现金盘点和解缴登记
**使用频率：** 每班次1次
**时间压力：** 中等

**优化后流程：**

```
[进入解缴] → [查看应收] → [输入实收] → [查看差额] → [选择原因(如有)] → [提交]
      ↓          ↓            ↓           ↓              ↓              ↓
 从交接或独立入口  自动计算    数字键盘    自动计算高亮    仅超阈值必填    成功反馈
```

**具体设计决策：**

| 步骤 | 用户行为 | 设计决策 | 依据 |
|------|---------|---------|------|
| 1 | 查看应收 | 大字号显示系统计算的应收现金 | 明确基准数字 |
| 2 | 输入实收 | 数字键盘输入，支持金额格式校验 | 减少输入错误 |
| 3 | 查看差额 | 自动计算并用颜色标识（红=短款，绿=长款） | 直观反馈 |
| 4 | 填原因 | 超过阈值（如±50元）时必填，提供常用原因选项 | 平衡效率和审计 |
| 5 | 提交 | 简洁的提交按钮，成功后显示确认信息 | 快速完成 |

### 任务 4：查看交接班历史 (US-016 ~ US-017)

## 3. 设计系统应用

### 3.1 页面结构

交接班管理模块包含以下主要页面：

| 页面 | 路由 | 说明 |
|------|------|------|
| 班次汇总 | `/shift-handover/summary` | 当前班次实时数据看板 |
| 交接班向导 | `/shift-handover/handover` | 步骤式交接班流程 |
| 现金解缴 | `/shift-handover/settlement` | 现金解缴登记 |
| 交接班历史 | `/shift-handover/history` | 历史记录列表 |
| 交接班详情 | `/shift-handover/history/:id` | 单条记录详情 |
| 解缴审核 | `/shift-handover/settlement-review` | 财务审核列表 |

### 3.2 导航结构

```
基础运营
├── 站点管理
├── 交接班管理
│   ├── 班次汇总      ← 默认页
│   ├── 交接班历史
│   └── 解缴审核      ← 仅财务/站长可见
├── 设备设施管理
└── 巡检/安检管理
```

### 3.3 颜色语义

| 颜色 | 使用场景 |
|------|----------|
| 绿色 (success) | 检查通过、长款、已完成 |
| 黄色 (warning) | 警告、待处理、需关注 |
| 红色 (error) | 检查失败、短款、异常 |
| 蓝色 (info) | 信息提示、进行中 |
| 灰色 (neutral) | 已取消、禁用状态 |

### 3.4 交互模式

| 模式 | 使用场景 | 说明 |
|------|----------|------|
| 步骤向导 (Stepper) | 交接班流程 | 分步骤引导，明确进度 |
| 卡片仪表盘 (Card Dashboard) | 班次汇总 | 指标分组展示 |
| 数据表格 (DataTable) | 历史记录列表 | 支持排序、筛选、分页 |
| 表单弹窗 (Modal Form) | 异常标注、原因填写 | 聚焦单一任务 |
| 确认对话框 (Confirm Dialog) | 提交交接、审核操作 | 重要操作双重确认 |

## 5. 可访问性设计

### 5.1 键盘导航

| 场景 | 快捷键 | 说明 |
|------|--------|------|
| 刷新数据 | `R` | 手动刷新班次汇总 |

- 所有金额字段需有 `aria-label` 描述单位
- 状态变化需使用 `aria-live` 通知
- 图标按钮需有文字描述

## 6. 错误处理策略

### 6.1 输入验证

| 校验类型 | 时机 | 反馈方式 |
|----------|------|----------|
| 必填校验 | 失焦/提交时 | 字段下方红色提示文字 |
| 格式校验 | 输入时 | 实时提示 |
| 业务校验 | 提交时 | 表单上方提示框 |

### 6.2 操作错误

| 场景 | 处理方式 |
|------|----------|
| 网络错误 | 显示重试按钮，保留用户输入 |
| 权限不足 | 禁用按钮 + 悬浮提示说明 |
| 数据冲突 | 弹窗提示，提供刷新或覆盖选项 |

### 6.3 系统错误

| 场景 | 处理方式 |
|------|----------|
| 服务不可用 | 显示友好错误页，提示联系管理员 |
| 超时 | 自动重试一次，仍失败则提示 |

## 7. 动效设计

### 7.1 动效原则

- **功能性**：动效服务于引导和反馈，非装饰
- **快速**：过渡时长200-300ms，避免耽误用户
- **一致性**：同类操作使用相同动效

### 7.2 动效应用

| 场景 | 动效 | 时长 |
|------|------|------|
| 页面切换 | 淡入淡出 | 200ms |
| 弹窗打开/关闭 | 缩放 + 淡入淡出 | 250ms |
| 数据刷新 | 加载旋转 + 淡入 | 300ms |
| 卡片展开 | 高度过渡 | 200ms |
| 状态切换 | 颜色过渡 | 200ms |
