## 1. 用户角色分析

### 1.1 收银员 (cashier)

**基本信息：**
- 典型画像：22-35 岁，高中或大专学历，经过岗位培训
- 工作环境：站点现场收银台，PC 或 POS 终端操作
- 使用频率：全天高频操作（30-100 笔/班次），订单查看/支付为核心动作
- 技术熟练度：中等偏低，依赖系统引导完成操作

**核心诉求：**
- 主要目标：快速完成收银支付、查找特定订单、处理简单异常
- 痛点：
  - 高峰期操作繁忙，需一键收银而非多步操作
  - 客户等待时间敏感，支付流程不能卡壳
  - 异常订单处理流程不清，需明确指引
- 期望：操作步骤少、反馈及时、错误可恢复

**场景约束：**
- 时间压力：极高——每笔交易需在 1-2 分钟内完成
- 注意力分散：高——需同时与客户沟通
- 专业知识：熟悉本站常用操作，不熟悉财务和审批流程

---

### 1.2 站长 (station_master)

**基本信息：**
- 典型画像：30-45 岁，持危化品经营许可证，熟悉站点全部业务
- 工作环境：站点现场办公室，PC 为主，偶尔使用移动端
- 使用频率：订单查看 5-10 次/日，异常处理 1-3 次/日，退款审批 每周数次
- 技术熟练度：中等

**核心诉求：**
- 主要目标：监控站点营业状况、处理异常订单、审批退款、手动补录订单
- 痛点：
  - 异常订单分散在不同位置，缺乏集中管理入口
  - 退款审批信息不完整，难以判断是否合理
  - 班次结算时需快速核对订单数据
- 期望：异常订单一目了然、审批信息充分、统计数据实时更新

**场景约束：**
- 时间压力：中等——异常处理需及时但非即时
- 注意力分散：高——经常被现场事务打断
- 专业知识：全面了解站点业务规则

---

### 1.3 财务人员 (finance)

**基本信息：**
- 典型画像：28-45 岁，财务/会计专业背景
- 工作环境：办公室，PC 端操作
- 使用频率：订单查询 3-5 次/日，导出 每周数次，月末集中使用
- 技术熟练度：中等，擅长数据处理

**核心诉求：**
- 主要目标：核对订单数据、查看退款记录、导出对账数据
- 痛点：
  - 订单数据需按多维度筛选（站点、时间、支付方式），操作复杂
  - 退款流水需与原订单关联，核对困难
  - 导出的数据需满足财务报表格式
- 期望：强大的筛选和搜索能力、清晰的退款关联、便捷的导出

**场景约束：**
- 时间压力：月末对账时压力大
- 注意力分散：低，专注数据核对
- 专业知识：深厚财务知识，对业务流程了解有限

---

### 1.4 运营经理 (ops_manager)

**基本信息：**
- 典型画像：35-50 岁，负责多站点运营管理
- 工作环境：总部办公室，PC 端
- 使用频率：订单监控 2-3 次/日，审批 每周数次
- 技术熟练度：中等偏高

**核心诉求：**
- 主要目标：多站点订单数据监控、异常订单审核、退款审批
- 痛点：
  - 多站点数据分散，无法快速对比
  - 审批待办容易遗漏
- 期望：全局视图、审批提醒醒目

## 2. 核心任务流程

### 任务 1：查看订单列表（US-001, US-002, US-003）

**用户目标：** 快速浏览和定位站点的订单记录
**使用频率：** 30+ 次/日（收银员）、5-10 次/日（站长）
**时间压力：** 高

**优化后流程：**

```
[进入订单管理] → [扫视统计概览] → [浏览订单列表] → [筛选/搜索] → [查看详情]
       ↓               ↓               ↓               ↓             ↓
  侧边栏一级入口   顶部统计卡片    时间倒序列表    顶部筛选栏      Drawer 展示
```

**具体设计决策：**

| 步骤 | 用户行为 | 设计决策 | 依据 |
|------|---------|---------|------|
| 1 | 进入页面 | 订单列表为订单管理落地页，顶部显示统计卡片（当日/当班维度切换）：总订单数、总金额、总充装量、支付方式分布 | 站长需 1 秒掌握经营状况 |
| 2 | 浏览列表 | 表格列：订单号、时间、枪号、燃料类型、数量(kg)、金额(元)、支付状态(Tag)、支付方式。时间倒序排列 | 最新订单最重要 |
| 3 | 筛选 | 顶部筛选栏：时间范围（DateRangePicker，默认今日）、枪号（Select）、燃料类型（Select）、支付状态（Select）、支付方式（Select） | 多维度筛选满足各角色需求 |
| 4 | 搜索 | 搜索框支持输入订单号/车牌号/会员手机号，输入后回车或实时搜索 | 精准定位特定订单 |
| 5 | 查看详情 | 点击行打开 Drawer，显示订单完整信息：基本信息、加注明细、支付明细、优惠明细、会员/车辆信息 | Drawer 不离开列表页上下文 |

### 任务 2：手动创建订单（US-007）

**用户目标：** 在管理后台手动创建一笔充装订单
**使用频率：** 每日数次（站长补录）
**时间压力：** 中等

**优化后流程：**

```
[订单列表页] → [点击"创建订单"] → [选择枪号] → [输入充装量] → [确认金额] → [提交]
      ↓              ↓               ↓             ↓             ↓          ↓
  右上角按钮     Drawer 打开    自动加载单价    实时计算金额   价格×数量    生成订单号
```

**具体设计决策：**

| 步骤 | 用户行为 | 设计决策 | 依据 |
|------|---------|---------|------|
| 1 | 发起创建 | 订单列表页右上角"创建订单"按钮（+ 图标），点击打开侧抽屉（Drawer） | 补充手段，入口明确但不过于突出 |
| 2 | 选择枪号 | Select 下拉加载当前站点枪列表，选中后自动显示枪关联的燃料类型和当前单价 | 枪→燃料类型→价格自动联动，减少手动输入 |
| 3 | 输入充装量 | 数字输入框（单位：kg），输入后实时计算：充装量 × 单价 = 订单金额 | 实时金额反馈，减少确认步骤 |
| 4 | 可选信息 | 可选填：车牌号（Input）、备注（TextArea）、标签（Select 多选） | 非必填字段折叠显示 |
| 5 | 提交 | 点击"创建订单"，系统生成订单号，订单状态为"待支付"。Drawer 关闭，列表刷新 | 创建后即可在列表中看到新订单 |

---

### 任务 3：支付操作（US-009）

**用户目标：** 为待支付订单选择支付方式并完成收银
**使用频率：** 30-100 次/班次
**时间压力：** 极高

**优化后流程：**

```
[待支付订单] → [点击"收银"] → [选择支付方式] → [确认支付] → [支付完成]
      ↓              ↓              ↓              ↓            ↓
  列表中橙色状态   弹出支付面板    大按钮选择      金额确认    Toast + 状态刷新
```

**具体设计决策：**

| 步骤 | 用户行为 | 设计决策 | 依据 |
|------|---------|---------|------|
| 1 | 发起收银 | 待支付订单行右侧显示"收银"主按钮（蓝色），点击弹出支付 Modal | 收银是最高频操作，按钮醒目 |
| 2 | 查看金额 | Modal 顶部大字号显示应付金额（元），下方显示订单摘要（枪号、燃料、数量） | 金额是核心信息，需最大化显示 |
| 3 | 选择方式 | 4 个支付方式大按钮（现金/微信/支付宝/银联），每个带图标和名称，网格布局 | 大按钮减少操作误差，图标辅助识别 |
| 4 | 现金支付 | 选择"现金"后，显示实收金额输入框 + 自动计算找零。快捷键 Enter 确认 | 收银员最常用，需极简流程 |
| 5 | 电子支付 | 选择微信/支付宝/银联后，显示"等待支付"倒计时（5分钟）+ 确认按钮。MVP 模拟回调 | 模拟第三方支付流程 |
| 6 | 支付完成 | Toast"支付成功"，Modal 关闭，订单状态刷新为"已支付"（绿色） | 即时反馈，无需手动刷新 |

### 任务 4：处理异常订单（US-012, US-013, US-014）

**用户目标：** 查看并处理异常订单（挂起/补单）
**使用频率：** 1-3 次/日
**时间压力：** 中等

**优化后流程：**

```
[异常订单页面] → [查看待处理列表] → [挂起/补单] → [审核（MVP+）]
       ↓               ↓                ↓              ↓
  侧边栏入口+Badge   按异常类型分组    操作 Drawer     审批列表
```

**具体设计决策：**

| 步骤 | 用户行为 | 设计决策 | 依据 |
|------|---------|---------|------|
| 1 | 发现异常 | 侧边栏"异常订单"菜单项旁显示红色 Badge（待处理数量），页面顶部统计卡片显示各类异常数量 | 异常不应被遗漏 |
| 2 | 浏览列表 | 表格显示：订单号、时间、异常类型(Tag)、异常原因、处理状态(Tag)。支持按异常类型和处理状态筛选 | 分类清晰，快速定位 |
| 3 | 挂起操作 | 待处理订单行有"挂起"按钮（灰色），点击后状态变更为"已挂起"（黄色标签），不参与班次结算 | 一键操作，不需要表单 |
| 4 | 补单操作 | 点击"补单"按钮，打开补单 Drawer：选择支付方式、输入实付金额、输入补单原因（必填）。提交后记录操作人 | 补单需完整信息，Drawer 适合 |
| 5 | 审核（MVP+） | 补单提交后进入"待审核"状态，站长/运营经理在审核列表中通过/驳回 | 与退款审批流程保持一致 |

---

### 任务 5：退款流程（US-016, US-017, US-018）

**用户目标：** 对已支付订单发起退款并完成审批
**使用频率：** 每周数次
**时间压力：** 中等

**优化后流程：**

```
[订单详情] → [发起退款] → [填写退款信息] → [提交申请] → [审批] → [退款完成]
      ↓           ↓             ↓              ↓           ↓          ↓
  "退款"按钮    Modal 弹出    选类型+金额+原因   进入审批队列  通过/驳回   状态更新
```

**具体设计决策：**

| 步骤 | 用户行为 | 设计决策 | 依据 |
|------|---------|---------|------|
| 1 | 发起退款 | 已支付/已完成订单详情页显示"退款"按钮（红色文字），点击弹出退款 Modal | 退款是敏感操作，按钮用红色文字（非主按钮）提示谨慎 |
| 2 | 填写信息 | Modal 中：退款类型 Radio（全额退款/部分退款）→ 退款金额（部分退款时可编辑，全额自动填入）→ 退款原因（必填 TextArea） | 信息完整性确保审批可判断 |
| 3 | 提交 | 二次确认"是否确认提交退款申请？"。确认后订单状态变更为"退款审批中" | 敏感操作二次确认 |
| 4 | 审批列表 | 站长在"退款审批"页面查看待审列表：原订单号、退款金额、退款原因、申请人、申请时间 | 与价格管理审批列表风格一致 |
| 5 | 审批操作 | "通过"按钮（绿色）+ "驳回"按钮（红色），驳回需填写原因 | 与 2.1 调价审批操作一致 |
| 6 | 退款记录 | 退款完成后，订单详情显示退款信息；财务可在退款记录页面汇总查看 | 退款记录关联原订单，方便核对 |

---

### 任务 6：订单导出（US-021）

**用户目标：** 按条件导出订单数据用于财务对账
**使用频率：** 每周数次
**时间压力：** 月末高

**优化后流程：**

```
[设置筛选条件] → [点击"导出"] → [选择格式] → [下载文件]
       ↓              ↓             ↓            ↓
  复用列表筛选     右上角按钮    Excel/CSV     浏览器下载
```

**具体设计决策：**

| 步骤 | 用户行为 | 设计决策 | 依据 |
|------|---------|---------|------|
| 1 | 设置范围 | 复用订单列表页的筛选条件作为导出范围 | 所见即所导，不需要额外配置 |
| 2 | 触发导出 | 筛选栏右侧"导出"按钮（下载图标） | 位置固定，易于发现 |
| 3 | 格式选择 | 默认导出为 Excel (.xlsx)，后续支持 CSV | Excel 是财务最常用格式 |
| 4 | 数据量提示 | 导出前提示数据条数，超过 5000 条建议缩小范围 | 防止大文件生成过慢 |

## 3. 页面结构概览

### 3.1 导航结构

```
能源交易
  └─ 订单管理
       ├─ 订单列表        ← 落地页：统计 + 订单列表（US-001~005）
       ├─ 异常订单        ← 异常订单列表 + 处理操作（US-012~015）
       ├─ 退款管理        ← 退款记录 + 退款审批（US-016~018）
       └─ 订单设置        ← 标签管理（US-019~020）
```

### 3.2 页面层次

| 层级 | 页面 | 核心内容 | 用户角色 |
|------|------|---------|---------|
| L1 | 订单列表 | 统计卡片 + 订单列表 + 筛选/搜索 | 全部角色 |
| L2 | 订单详情 Drawer | 基本信息 + 加注明细 + 支付明细 + 操作按钮 | 全部角色 |
| L2 | 创建订单 Drawer | 枪选择 + 充装量 + 金额计算 | 站长 |
| L2 | 支付面板 Modal | 金额 + 支付方式 + 确认 | 收银员 |
| L1 | 异常订单 | 统计 + 异常订单列表 + 筛选 | 站长、运营经理 |
| L2 | 补单 Drawer | 支付信息 + 原因 | 站长 |
| L1 | 退款管理 | Tab(退款记录/退款审批) | 站长、财务 |
| L2 | 退款申请 Modal | 退款类型 + 金额 + 原因 | 收银员 |
| L1 | 订单设置 | 标签管理列表 | 站长 |

## 4. 交互规范

### 4.1 操作反馈

| 操作类型 | 反馈方式 | 示例 |
|---------|---------|------|
| 支付成功 | message.success + 列表刷新 | "支付成功" |
| 退款提交 | message.info + 状态更新 | "退款申请已提交，等待审批" |
| 审批通过 | message.success + 列表移除该项 | "审批通过，退款已执行" |
| 审批驳回 | message.warning | "已驳回，原因：金额不符" |
| 补单成功 | message.success + 状态更新 | "补单成功" |

## 5. 与已有模块的一致性设计

### 5.1 沿用模式

| 设计模式 | 来源模块 | 本模块应用 |
|---------|---------|-----------|
| 顶部统计卡片 + 下方列表 | 1.4 巡检管理、2.1 价格管理 | 订单列表页、异常订单页 |
| 详情 Drawer（不跳转） | 2.1 价格管理（调价历史） | 订单详情 Drawer |
| 审批列表 + 通过/驳回 | 2.1 价格管理（调价审批） | 退款审批、补单审核 |
| 侧边栏 Badge 徽章 | 1.4 巡检（问题记录）、2.1（审批） | 异常订单待处理数量 |
| Tab 切换视图 | 1.3 设备管理（维保工单状态 Tab） | 退款管理（记录/审批 Tab） |
| 全局站点选择器 | 全模块共用 | 通过 useOutletContext 消费 |

### 5.2 新增模式

| 设计模式 | 应用场景 | 说明 |
|---------|---------|------|
| 支付面板 Modal | 收银操作 | 大按钮选择支付方式 + 金额确认 |
| 状态流转时间线 | 订单详情 | Steps 组件显示订单生命周期 |
| 实时金额计算 | 创建订单 Drawer | 枪选择 → 价格自动获取 → 数量输入 → 金额实时计算 |
