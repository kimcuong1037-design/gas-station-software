# 项目合作原则 (Project Constitution)

本文件记录项目合作的最高准则，所有参与者（包括 AI Agent）必须遵守。

---

## 原则一：拆解优先，不直接跳入解决方案

- 面对大的、复杂的问题，必须先拆解为可管理的子问题
- 禁止在未充分理解问题全貌的情况下直接编码实现
- 每个子问题应有明确的输入、输出和验收标准

## 原则二：规范与对齐优先于编码

- 花最多的时间在：建立规范、对齐项目规划、确保需求理解一致、拆解需求、进行设计
- 用户将在规范和设计阶段深度参与，在编码实现阶段减少参与
- 编码前必须有经过确认的设计方案

## 原则三：忠实、准确

- 分析、设计和实现必须忠实于已确认的需求和规范
- 不臆造、不假设、不跳过
- 数据和结论必须有据可查

## 原则四：先框架后细节，先 MVP 后优化

- 先搭建产品整体框架，实现 MVP 原型
- 再优化细节以及添加创新或差异化功能
- 避免过早优化和过度设计

## 原则五：不确定时必须确认

- 当存在定义不清楚的概念或问题时，必须与用户确认
- 不做未经确认的假设
- 宁可多问一次，不可做错一步

## 原则六：修正经验持续积累

- 每次用户对输出进行修正时，将修正的背景及所获得的经验记录到独立文件
- **精简版**（日常引用）：`docs/CORRECTIONS.md` — 按主题归纳的模式速查表 + 摘要表
- **完整归档**：`docs/CORRECTIONS-ARCHIVE.md` — 每条记录的完整修正内容、原因分析和经验总结
- 新增记录时：在 CORRECTIONS.md 摘要表追加一行 + 关联模式编号；在 CORRECTIONS-ARCHIVE.md 补充完整上下文
- 方便后续访问与调用，避免重复犯错

## 原则七：文件删除必须双重确认

- **绝对禁止**未经确认删除任何文件或文件夹
- 任何涉及删除文件/文件夹的操作，必须进行**至少两次独立的用户确认**：
  1. 第一次：明确告知用户将要删除的文件/文件夹列表，获得确认
  2. 第二次：在执行前再次确认，提醒“此操作不可撤销”
- 优先采用**非破坏性方案**：新建目录而非覆盖、移动而非删除、软删除而非硬删除
- 使用外部工具（如 `create-vite`、`npm init`）时，必须检查其是否会删除现有文件，并在交互式提示中选择保留现有文件的选项
- 此原则适用于整个项目生命周期

## 原则八：关键文档修改须用户审批后提交

- 以下类型的文档修改，**必须在 commit 前获得用户明确 approve**：
  - **计划类**：PLAN.md、PLAN-CN.md、ROADMAP.md、PROGRESS.md
  - **架构类**：architecture.md（任何模块）
  - **需求类**：requirements.md、user-stories.md（任何模块）
  - **Skills 定义**：docs/skills/ 下的所有 Skill 文件
  - **协作规范**：CONSTITUTION.md、STANDARDS.md、AGENT-PLAN.md、CLAUDE.md、SESSION-PROTOCOL.md
- 工作流程：完成修改 → 向用户展示变更内容 → 用户 approve → 执行 git commit
- 纯前端代码修改（组件、样式、i18n、mock 数据等）不受此限制，可直接提交
- 此原则旨在确保项目方向性决策始终由用户把控
