## §1 认领规则

1. **认领前沟通** — 通过团队约定的沟通渠道告知团队你计划认领哪个模块
2. **在 §2 表中登记** — 填入你的名字，状态改为 `🔒 已认领`，记录认领日期
3. **模块互斥** — 同一模块同一时间只允许一人开发，避免文件冲突
4. **3 日启动限期** — 认领后 3 个工作日内必须提交第一个 PR（通常是 `requirements.md`），否则认领自动释放
5. **完成时更新** — 模块完成后状态改为 `✅ 已完成`，注明 UI 评审分数
6. **释放认领** — 如果因故无法继续，及时将状态改回 `☐ 待认领` 并通知团队

**状态枚举：**

| 状态 | 含义 |
|------|------|
| ☐ 待认领 | 无人开发，可认领 |
| 🔒 已认领 | 已有人认领，尚未开始编码 |
| 🔧 开发中 | 文档或代码开发进行中 |
| 📋 评审中 | UI 评审 / PR review 进行中 |
| ✅ 已完成 | 交付完成，标注 UI 评分 |

## §3 共享文件修改协议

以下文件被多个模块共享。**修改前必须 `git pull`**，修改后**尽快 commit + push**，不要长时间持有未提交的变更。

#### 前端共享文件

| 文件 | 冲突风险 | 修改协议 |
|------|---------|---------|
| `frontend/src/router.tsx` | 🔴 高 | 只新增自己模块的路由区块；不要重新排序已有路由 |
| `frontend/src/components/AppLayout.tsx` | 🔴 高 | 只新增自己模块的菜单区块；不要修改其他模块的菜单项 |
| `frontend/src/components/RequirementTag.tsx` | 🟡 中 | 只新增自己模块的 import + mapping 合并 |
| `frontend/src/locales/zh-CN/index.ts` | 🟡 中 | 只新增自己模块的 namespace；不修改已有 namespace 的内容 |
| `frontend/src/locales/en-US/index.ts` | 🟡 中 | 同上 |
| `docs/cross-module-erd.md` | 🟢 低 | 只新增自己模块的章节（§3.x, §4.x） |
| `docs/STANDARDS.md` §1 术语表 | 🟢 低 | 新增术语需通过团队约定的沟通渠道通知全员 |

#### 后端共享文件

| 文件 | 冲突风险 | 修改协议 |
|------|---------|---------|
| `backend/app/models/__init__.py` | 🔴 高 | 只新增自己模块 Model 的 import；不修改其他模块的 Model |
| `backend/app/api/__init__.py` | 🔴 高 | 只新增自己模块 Blueprint 的注册；不修改已有注册 |
| `backend/migrations/` | 🟡 中 | 迁移文件由 `flask db migrate` 自动生成，commit 前 `git pull` 确认无冲突 |
| `backend/tests/conftest.py` | 🟡 中 | 只新增自己模块的 fixture；不修改已有 fixture |
| `backend/requirements.txt` | 🟢 低 | 新增依赖需通知团队，避免版本冲突 |

### 冲突处理

如果 `git pull` 后发现冲突：
1. 只解决**自己模块相关的区块**，保留他人的变更
2. 解决后运行 `npm run build` 确认编译通过
3. 如果无法判断他人变更的意图，通过团队约定的沟通渠道联系对方确认

## §4 Git 分支与 PR 流程

### 分支策略

| 分支类型 | 命名格式 | 说明 |
|---------|---------|------|
| `main` | — | 稳定分支，只接受 PR 合入，**禁止直接 push** |
| `feature/<module-id>-<desc>` | `feature/7.1-data-analytics` | 模块开发分支 |
| `fix/<desc>` | `fix/scroll-x-missing` | 修复分支 |
| `docs/<desc>` | `docs/update-standards` | 纯文档变更分支 |

### PR 创建流程

```bash
# 1. 从 main 创建 feature branch
git checkout main && git pull origin main
git checkout -b feature/<module-id>-<desc>

# 2. 开发过程中定期同步 main
git fetch origin main
git rebase origin/main

# 3. 完成后推送并创建 PR
git push -u origin feature/<module-id>-<desc>
# 在 GitHub/GitLab 创建 PR
```

### 分层审批

| PR 内容 | 审批人 | 说明 |
|---------|--------|------|
| **关键文档** | **Roger** | requirements / architecture / ux-design / ui-schema / STANDARDS / 治理文档 |
| **纯代码** | 任意团队成员 | 页面组件 / i18n / mock / router 等 |
| **混合 PR** | **Roger** | 按最高等级——含关键文档则由 Roger 审批 |

### PR 描述模板

创建 PR 时请在描述中注明：

```markdown
## 变更内容
- 对应步骤：AGENT-PLAN Step X（前端）/ BE Step X（后端）
- 模块：<module-id> <module-name>
- 类型：[ ] 前端  [ ] 后端  [ ] 混合

## 包含文档类型
- [ ] 关键文档（需 Roger approve）
- [ ] 纯代码（团队互审）

## 影响的前端共享文件
- [ ] router.tsx
- [ ] AppLayout.tsx
- [ ] RequirementTag.tsx
- [ ] i18n (zh-CN / en-US)
- [ ] 无前端共享文件变更

## 影响的后端共享文件（如有）
- [ ] backend/app/models/__init__.py
- [ ] backend/app/api/__init__.py
- [ ] backend/migrations/（迁移文件）
- [ ] backend/tests/conftest.py
- [ ] 无后端共享文件变更

## 后端交付 Checklist（后端 PR 必填）
- [ ] SQLAlchemy Models 与 architecture.md §DB Schema 一一对应
- [ ] 迁移文件已创建且 `flask db upgrade` 可执行
- [ ] 所有 API 端点已实现（方法 + 路径 + 响应结构）
- [ ] pytest 测试套件通过
- [ ] Swagger UI 文档正确
```

### 合入方式

- **Squash merge** 到 main — 保持 main 分支历史简洁
- 合入后删除 feature branch
- 通过团队约定的沟通渠道通知团队已合入
