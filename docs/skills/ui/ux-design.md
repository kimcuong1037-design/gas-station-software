# Skill: 用户体验设计 (UX Design)

## 元信息

- **Skill ID:** `ux-design`
- **所属 Agent:** UI 设计 Agent (UI Designer)
- **输入：**
  - User Story 文档（`docs/features/<domain>/<module>/user-stories.md`，由 `user-story-writing` Skill 产出）
  - 行业知识库（`knowledge-base/` 目录下的所有文件，理解用户角色、工作场景、业务流程）
  - UI/UX 设计规范（`docs/STANDARDS.md` §4）
- **输出：**
  - UX 设计文档（存放于 `docs/features/<domain>/<module>/ux-design.md`）
- **依赖：** `user-story-writing` Skill 的产出
- **下游：** `ui-schema-design` Skill（UX 设计文档作为其输入之一）

## 核心理念

### 设计原则

1. **用户视角优先**：始终从用户的角度思考，而非系统的角度
2. **最小操作原则**：能一步完成的操作，绝不设计成两步
3. **认知负荷最小化**：减少用户需要记忆、理解、决策的信息量
4. **渐进式披露**：先展示核心功能，高级功能按需展开
5. **容错设计**：预防错误发生，提供清晰的错误恢复路径
6. **一致性**：相同功能在不同页面保持一致的交互模式

### 可用性目标

| 维度 | 目标 | 衡量方式 |
|------|------|---------|
| 可学习性 | 新用户 10 分钟内完成核心任务 | 首次使用成功率 |
| 效率 | 高频操作 ≤3 次点击完成 | 任务完成步骤数 |
| 可记忆性 | 用户间隔一周后仍能记住操作方式 | 二次使用成功率 |
| 错误率 | 关键操作错误率 <1% | 错误日志统计 |
| 满意度 | 用户满意度评分 ≥4/5 | 用户调研 |

## 流程定义

### 步骤 1: 用户角色分析

基于 User Story 和行业知识库，深入理解目标用户：

**用户画像模板：**

```markdown
### 角色：{{角色名称}}

**基本信息：**
- 典型年龄/技术熟练度：{{描述}}
- 工作环境：{{办公室/现场/移动端}}
- 使用频率：{{每日多次/每日一次/每周几次}}

**核心诉求：**
- 主要目标：{{用户想要达成什么}}
- 痛点：{{当前方式的问题}}
- 期望：{{理想的使用体验}}

**场景约束：**
- 时间压力：{{是否需要快速操作}}
- 注意力分散：{{是否会被其他事情打断}}
- 专业知识：{{需要什么背景知识}}
```

### 步骤 2: 任务流程分析

为每个核心 User Story 绘制用户任务流：

**任务流分析模板：**

```markdown
### 任务：{{任务描述}}（对应 US-XXX）

**用户目标：** {{用户想要达成的结果}}

**当前流程（如有）：**
1. {{步骤1}} — 痛点：{{问题}}
2. {{步骤2}} — 痛点：{{问题}}
...

**优化后流程：**
1. {{步骤1}} — 设计要点：{{为什么这样设计}}
2. {{步骤2}} — 设计要点：{{为什么这样设计}}
...

**关键决策点：**
- {{用户需要做的决策}} → 设计支持：{{如何帮助用户决策}}
```

### 步骤 2.5: 端到端用户旅程走查

在完成单个任务流分析后，进行完整的端到端走查——模拟用户从"打开系统"到"完成核心目标"的全过程，验证流程的完整性和上下文连贯性。

**走查方法：**

1. **确定起点**：用户打开系统后看到的第一个页面是什么？用户身份是否明确？
2. **逐步推进**：用户要完成核心任务（如发起交接班），需要经过哪些页面？每一步的前置条件是什么？
3. **检查上下文**：每个页面需要的上下文数据（如"当前站点""当前班次""值班人员"）是否已在前序步骤中建立？
4. **发现断点**：是否存在"前置数据没有入口获取"的情况？例如：需要排班数据但没有排班管理入口。
5. **跨模块依赖**：本模块依赖的外部数据（如站点信息、员工列表、排班计划），用户是否能在本模块中便捷访问？如果不能，是否需要在本模块提供入口或快捷链接？

**走查输出模板：**

```markdown
### 端到端走查：{{核心任务名称}}

| 步骤 | 用户行为 | 前置条件 | 数据来源 | 是否有入口 | 问题 |
|------|---------|---------|---------|-----------|------|
| 1 | 进入系统 | 用户已登录 | 认证模块 | ✅/❌ | {{问题描述}} |
| 2 | 选择站点 | 站点列表可用 | 站点管理 | ✅/❌ | |
| ... | | | | | |
```

**发现问题时：**
- 将缺失的环节记录为"流程断点"，提出具体的补救方案
- 如需在本模块增加外部数据的 UI 入口，建议以"子菜单"或"快捷链接"形式提供
- 将问题清单提交给用户确认后再继续设计

### 步骤 3: 操作效率优化

分析并优化操作步骤数：

**3.1 高频操作识别**

列出使用频率最高的操作（每日多次执行的任务），确保这些操作：
- 入口明显，无需寻找
- 操作步骤最少
- 支持快捷操作（键盘、批量等）

**3.2 点击次数审计**

| 任务 | 当前步骤数 | 优化后步骤数 | 优化方案 |
|------|-----------|-------------|---------|
| {{任务}} | {{N}} | {{M}} | {{具体方案}} |

**常见优化策略：**

| 策略 | 适用场景 | 示例 |
|------|---------|------|
| 智能默认值 | 大多数情况下值相同 | 日期默认今天、站点默认常用站点 |
| 记住上次选择 | 用户习惯固定 | 记住上次选择的筛选条件 |
| 批量操作 | 重复性操作 | 批量审批、批量导出 |
| 行内编辑 | 简单修改 | 直接在表格中编辑，无需打开弹窗 |
| 快捷键 | 高频操作 | Ctrl+S 保存，Enter 确认 |
| 拖拽操作 | 排序、分组 | 拖拽调整顺序 |
| 自动保存 | 表单填写 | 防止数据丢失 |

### 步骤 4: 导航与信息架构设计

**4.1 导航层级设计**

**原则：**
- 关键功能 ≤2 级可达（从首页到目标页面最多 2 次点击）
- 导航层级不超过 3 级
- 相关功能就近分组

**导航设计模板：**

```markdown
### 导航结构

**一级导航（侧边栏）：**
- {{菜单1}} — 用户心理模型：{{用户如何理解这个分类}}
- {{菜单2}} — 用户心理模型：{{...}}

**二级导航：**
- {{父菜单}} > {{子菜单}} — 访问频率：{{高/中/低}}

**快捷入口：**
- {{功能}} — 位置：{{顶部导航栏/首页卡片/...}}
```

**4.2 信息层次设计**

根据重要性和使用频率组织页面信息：

| 层次 | 内容类型 | 视觉处理 |
|------|---------|---------|
| 主要 | 用户最关注的核心数据 | 突出显示、大字号、强调色 |
| 次要 | 辅助信息、补充说明 | 正常显示、标准字号 |
| 三级 | 低频查看的详细信息 | 折叠/展开、灰色文字 |

### 步骤 5: 引导与帮助设计

**5.1 新手引导**

| 引导类型 | 适用场景 | 实现方式 |
|---------|---------|---------|
| 空状态引导 | 首次使用/无数据 | 空状态图 + 引导文字 + 操作按钮 |
| 步骤引导 | 复杂流程首次执行 | 高亮提示 + 步骤说明 |
| 功能发现 | 新功能上线 | 气泡提示 + "不再提示"选项 |
| 工具提示 | 字段/图标含义不明确 | Tooltip 悬停说明 |

**5.2 上下文帮助**

为可能产生困惑的地方提供帮助：

```markdown
### 帮助点设计

| 位置 | 用户困惑 | 帮助内容 | 呈现方式 |
|------|---------|---------|---------|
| {{字段/按钮}} | {{用户可能的疑问}} | {{解答}} | {{Tooltip/说明文字/链接}} |
```

**5.3 说明文字规范**

- **标签文字**：简洁明确，使用用户熟悉的词汇
- **占位符文字**：给出输入示例，而非重复标签
- **帮助文字**：解释"为什么"和"怎么做"，而非"是什么"
- **错误提示**：说明问题 + 给出解决方案

### 步骤 6: 错误预防与恢复设计

**6.1 错误预防**

| 策略 | 实现方式 | 示例 |
|------|---------|------|
| 约束输入 | 使用选择器代替自由输入 | 下拉选择而非手动输入 |
| 实时校验 | 输入时即时反馈 | 输入格式错误立即提示 |
| 确认危险操作 | 二次确认 | 删除前弹窗确认 |
| 操作预览 | 显示操作结果预览 | 批量操作前显示影响范围 |

**6.2 错误恢复**

| 场景 | 恢复方案 |
|------|---------|
| 误删除 | 提供撤销功能或回收站 |
| 表单填写中断 | 自动保存草稿 |
| 操作失败 | 保留用户输入，提供重试 |
| 网络断开 | 离线缓存，网络恢复后同步 |

### 步骤 7: 反馈与状态设计

**系统反馈设计：**

| 操作类型 | 反馈方式 | 时机 |
|---------|---------|------|
| 点击按钮 | 按钮状态变化（loading） | 立即 |
| 提交成功 | 成功提示 + 结果展示 | 操作完成后 |
| 提交失败 | 错误提示 + 保留数据 | 操作失败后 |
| 长时间操作 | 进度条/进度百分比 | 持续显示 |
| 后台任务 | 状态更新通知 | 任务完成时 |

**状态可见性：**
- 当前位置：面包屑、高亮当前菜单
- 操作状态：加载中、已选中、已禁用
- 数据状态：空状态、错误状态、加载状态

## 附录：UX 设计模式参考

### A. 常见交互模式

| 模式 | 适用场景 | 优点 |
|------|---------|------|
| 渐进式披露 | 复杂表单/设置 | 降低初始认知负荷 |
| 分步向导 | 多步骤流程 | 清晰的进度感 |
| 即时预览 | 设置/样式调整 | 所见即所得 |
| 撤销/重做 | 编辑操作 | 鼓励探索 |
| 自动保存 | 长表单 | 防止数据丢失 |
| 模糊搜索 | 大量选项 | 快速定位 |
| 最近使用 | 历史记录 | 快速重复操作 |
| 收藏/固定 | 常用项目 | 快速访问 |

### B. 认知负荷优化策略

| 策略 | 说明 |
|------|------|
| 分块呈现 | 将信息分组，每组 5±2 项 |
| 视觉层次 | 用大小、颜色、间距区分重要性 |
| 相似性分组 | 相关功能视觉相近 |
| 一致性 | 相同功能保持相同交互方式 |
| 熟悉模式 | 采用用户已熟悉的交互模式 |
| 减少记忆 | 识别而非回忆（显示选项而非要求输入） |

### C. 加气站业务场景特殊考量

| 场景 | UX 考量 |
|------|---------|
| 加气员操作 | 可能在现场使用，需要大按钮、简洁界面、支持手套操作 |
| 财务对账 | 数据准确性关键，需要清晰的核对流程和差异提示 |
| 设备监控 | 异常状态需要醒目提示，支持快速响应 |
| 库存管理 | 低库存预警，批量操作支持 |
| 审批流程 | 状态流转清晰，待办事项突出 |
