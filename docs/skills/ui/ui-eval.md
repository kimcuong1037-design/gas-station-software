# UI 评估技能

使用结构化、可重复的框架评估 AI/LLM 生成的企业软件项目 UI。此技能生成可操作的评估报告，包含评分、发现和建议。

## 使用场景

- 审查由 LLM 生成的 UI 组件或界面
- 比较两个版本的 AI 生成 UI（A/B 评估）
- 在交付客户之前审计一批生成的界面
- 在 AI 驱动的 UI 生成流水线中建立质量门禁
- 评估 AI 生成的 UI 是否符合企业标准

## 评估框架

### 六大评估维度

每个 AI 生成的 UI 都需从六个维度进行评估。每个维度都有具体的评估标准和评分规则。

#### 1. 视觉保真度与设计一致性（权重：20%）

检查 UI 是否符合目标设计系统和品牌规范。

**评估标准：**
- 色彩方案遵循度（主色、辅色、强调色、语义色）
- 排版一致性（字体家族、字号、字重、行高）
- 间距和布局网格对齐（外边距、内边距、间隙）
- 组件样式一致性（按钮、输入框、卡片是否匹配设计令牌）
- 图标和图像风格协调性
- 响应式断点行为

**评分标准：**
- 5：与设计系统完美匹配，无偏差
- 4：轻微偏差（1-2 处不一致），易于修复
- 3：存在明显偏差，但整体外观可接受
- 2：显著的样式偏离，多个组件不符合品牌规范
- 1：与目标设计系统不相似

#### 2. 功能正确性（权重：25%）

验证 UI 在所有状态和交互中是否按预期工作。

**评估标准：**
- 正常流程工作正确
- 错误状态渲染和行为正确
- 空状态/零状态得到处理
- 存在加载/骨架屏状态
- 表单验证有效（客户端）
- 导航和路由正确
- 数据绑定和状态管理正常
- 边界情况：长文本、特殊字符、大数据集

**评分标准：**
- 5：所有状态都已处理，无功能缺陷
- 4：存在轻微边界情况问题，核心流程完美运行
- 3：某些状态缺失，但主要功能正常
- 2：核心功能存在缺陷或状态缺失
- 1：根本无法运行，功能失效

#### 3. 无障碍性（权重：20%）

确保 UI 可被残障人士使用，并符合 WCAG 标准。

**评估标准：**
- 语义化 HTML 结构（标题、地标、列表）
- 必要时使用 ARIA 标签和角色
- 键盘导航（Tab 顺序、焦点管理、焦点可见）
- 颜色对比度（WCAG AA 最低要求：正文 4.5:1，大文本 3:1）
- 屏幕阅读器兼容性
- 触摸目标尺寸（最小 44x44px）
- 图像和有意义内容的替代文本
- 不仅依赖颜色传达信息

**评分标准：**
- 5：符合 WCAG 2.1 AA 标准，键盘和屏幕阅读器支持优秀
- 4：存在轻微问题（1-2 个标签缺失），整体可访问
- 3：存在若干无障碍缺口，但基本导航可用
- 2：对辅助技术用户存在显著障碍
- 1：键盘或屏幕阅读器用户无法访问

#### 4. 代码质量与可维护性（权重：15%）

评估人类开发者是否能够理解、维护和扩展代码。

**评估标准：**
- 组件分解（单一职责、合理大小）
- 命名规范（描述性、一致性）
- 无硬编码魔法值（使用常量或设计令牌）
- TypeScript 类型（如适用）正确且有用
- 无死代码或注释掉的代码块
- 适当的关注点分离（逻辑与展示）
- 遵循框架惯用法（React hooks 正确使用等）
- 无明显安全问题（XSS、注入）

**评分标准：**
- 5：生产就绪，整洁、惯用、结构良好
- 4：质量良好，存在轻微风格问题
- 3：功能正常但需要重构以便长期维护
- 2：结构不佳，需要大量清理工作
- 1：无法维护，代码混乱，存在安全问题

#### 5. 性能（权重：10%）

检查 UI 是否高效加载和运行。

**评估标准：**
- 无不必要的重渲染（React Profiler 或等效工具）
- 高效的数据获取模式（无瀑布请求、适当缓存）
- 打包体积影响（不为单个工具函数导入整个库）
- 图像优化（懒加载、正确格式、尺寸适当）
- 动画性能（使用 CSS transform，不触发布局重排）
- Lighthouse 性能评分（如可测量）

**评分标准：**
- 5：已优化，无性能问题
- 4：存在轻微低效，对用户无感知影响
- 3：存在某些性能问题，可能在负载下影响用户体验
- 2：存在明显卡顿、加载缓慢或资源使用过度
- 1：严重性能问题，在较慢设备上无法使用

#### 6. 用户体验与交互设计（权重：10%）

评估视觉正确性之外的用户体验质量。

**评估标准：**
- 交互反馈（悬停状态、激活状态、过渡效果）
- 整个应用中交互模式的一致性
- 动效和动画的恰当使用
- 清晰的信息层级和视觉流
- 有帮助的错误信息（可操作，非技术性）
- 直观的表单布局和输入模式

**评分标准：**
- 5：精致、令人愉悦的交互，出色的用户体验
- 4：用户体验良好，存在轻微粗糙之处
- 3：可用但感觉通用或不够精致
- 2：交互令人困惑或缺少反馈
- 1：用户体验差，会让用户感到沮丧

## 评估流程

### 步骤 1：建立基准

评估前，收集以下上下文：

- **设计系统参考** — 设计令牌、组件库文档、Figma/Sketch 文件
- **功能需求** — UI 应该做什么（用户故事、验收标准）
- **目标无障碍标准** — 企业项目通常为 WCAG 2.1 AA
- **技术栈** — 框架、语言、关键库
- **客户特定需求** — 客户的任何额外标准

如果以上任何内容不可用，请在报告中注明为限制条件。

### 步骤 2：自动化检查

首先运行自动化工具以捕获容易发现的问题：

	无障碍性：  axe-core、Lighthouse 无障碍审计
	性能：      Lighthouse 性能、React Profiler
	代码质量：  ESLint、TypeScript 编译器（严格模式）
	视觉：      Chromatic/Percy（如果存在设计系统 Storybook）

在报告中记录自动化发现。

### 步骤 3：人工审查

针对每个维度进行人工审查，重点关注自动化遗漏的内容：

- 纯键盘导航测试（Tab、Shift+Tab、Enter、Escape、方向键）
- 屏幕阅读器抽查（VoiceOver、NVDA）
- 边界情况测试（空输入、超长文本、快速点击）
- 设计系统对比（与参考组件并排比较）
- 可维护性和模式的代码审查

### 步骤 4：评分与记录

针对每个维度：
1. 分配评分（1-5）
2. 列出具体发现（问题是什么、在哪里）
3. 按严重程度分类发现：**严重** / **重要** / **轻微** / **备注**
4. 为每个发现提供具体的修复建议

### 步骤 5：计算加权总分

	总分 = (视觉 × 0.20) + (功能 × 0.25) + (无障碍 × 0.20) 
	     + (代码 × 0.15) + (性能 × 0.10) + (用户体验 × 0.10)

**通过/失败阈值：**
- **可发布**（≥ 4.0）：可投入生产，只需轻微修复
- **修复后重新发布**（3.0 – 3.9）：交付前需要针对性改进
- **重做**（< 3.0）：存在重大问题，需重新生成或手动重建

### 步骤 6：生成报告

生成结构化的评估报告。完整模板请参见 `references/report-template.md`。

## 比较两个 UI 输出（A/B 评估）

比较两个版本时（例如不同提示词、不同模型、修改前后）：

1. 使用上述完整框架独立评估每个版本
2. 为每个维度创建并排比较表
3. 识别每个维度中哪个版本更强以及原因
4. 给出获胜者及其理由
5. 注明最佳结果是否可能是混合方案（例如，版本 A 的布局 + 版本 B 的无障碍性）

---

## 将结果反馈到生成过程

评估应改进未来的生成。每次评估周期后：

1. **模式目录** — 跟踪反复出现的问题（例如，"LLM 总是忘记模态框的焦点管理"）
2. **提示词改进** — 添加针对反复失败问题的具体说明
3. **参考示例** — 将高分输出添加为少样本示例
4. **防护措施** — 为最常见的失败建立自动化预检查

提示词改进工作流程请参见 `references/feedback-loop.md`。

## 定制化

### 调整权重

不同项目可能需要不同的权重分配：

- **医疗/政务**：将无障碍性提高到 30%，将用户体验降低到 5%
- **面向消费者的 SaaS**：将用户体验提高到 20%，将代码质量降低到 10%
- **内部工具**：将功能性提高到 30%，将视觉降低到 10%

### 添加自定义维度

如果客户有特定需求（例如国际化就绪、主题支持），可将其添加为第 7 个维度并设置适当权重，相应减少其他维度的权重。

### 严重程度定义

- **严重（Critical）**：阻止部署。安全漏洞、功能完全失效、违反 WCAG A 级标准。
- **重要（Major）**：发布前必须修复。重大功能缺陷、违反 WCAG AA 级标准、设计系统破坏。
- **轻微（Minor）**：应当修复。外观问题、轻微不一致、优化机会。
- **备注（Note）**：可有可无。润色建议、替代方案、未来改进。
