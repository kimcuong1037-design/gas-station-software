# Skill: 术语表维护 (Glossary Management)

## 元信息

- **Skill ID:** `glossary-management`
- **所属 Agent:** 需求分析 Agent (Requirement Analyst)
- **输入：**
  - 目标模块的文档集（`docs/features/<domain>/<module>/` 下的 5 份文档：requirements.md, user-stories.md, architecture.md, ux-design.md, ui-schema.md）
  - 术语规范（`docs/STANDARDS.md` §1 术语表）
  - 前端代码（如已存在）：types.ts, constants.ts, i18n key, 组件名, 路由定义
- **输出：**
  - 术语一致性报告（内嵌于执行日志，不单独生成文件）
  - STANDARDS.md §1 更新（如有新增术语）
  - 文档/代码修正（如有不一致）
- **依赖：** 至少一份模块文档已存在（通常在文档套件完成后执行）

---

## 适用场景

本 Skill 在以下时机触发：

| 触发时机 | 说明 |
|---------|------|
| **文档阶段出口** | 5 份文档全部完成、进入前端实现前，执行全量术语扫描 |
| **模块交付检查** | Step 12i Checklist 中的术语合规验证项 |
| **新子系统启动** | 首次进入新的 L1 子系统时，预扫描需求文档中的新术语并注册 |
| **用户主动触发** | 用户要求执行术语检查时 |

## 流程定义

### 步骤 1: 加载术语基准

从 `docs/STANDARDS.md` §1 加载当前术语表，构建术语查找索引：
- §1.1 核心业务术语：`中文术语 → { 英文, 代码命名, 说明 }`
- §1.2 角色术语
- §1.3 模块与菜单术语
- §1.4 状态/枚举术语
- 术语使用规则（如"加注机"泛称规则、"新增 vs 新建"规则）

### 步骤 2: 扫描目标文档

遍历目标模块的文档集，提取所有出现的：
- **实体名称**：页面名称、组件名、数据实体名
- **操作动词**：新增/新建/创建、编辑/修改、删除/移除
- **状态词**：各类状态枚举的中文表述
- **角色引用**：文档中提到的用户角色
- **菜单/导航命名**：侧边栏、面包屑、页面标题

扫描范围（按文档类型）：

| 文档 | 重点扫描内容 |
|------|-------------|
| requirements.md | 功能点名称(F001~)、功能描述中的实体和操作术语 |
| user-stories.md | US 标题、验收标准中的页面引用、角色名称 |
| ux-design.md | 任务流程标题、导航结构、页面层次表 |
| ui-schema.md | 页面名称(P01~)、组件名、路由、侧边栏文案、面包屑、i18n key |
| architecture.md | 实体名称、API 端点命名、枚举值 |

### 步骤 3: 比对与分类

将扫描结果与 STANDARDS.md 基准比对，输出分类结果：

| 分类 | 说明 | 处理方式 |
|------|------|---------|
| **一致** | 文档用词与 STANDARDS.md 完全匹配 | 无需处理 |
| **不一致** | 同一概念在不同文档/位置使用了不同名称 | 统一为 STANDARDS.md 规定的术语 |
| **新增术语** | 文档中出现了 STANDARDS.md 未收录的术语 | 评估后注册到 STANDARDS.md |
| **违反使用规则** | 如混用"加油机/加气机"、使用"新建"而非"新增" | 按规则修正 |

### 步骤 4: 跨文档命名一致性检查

针对同一 UI 元素，检查命名在 5 份文档间是否一致：

```
检查链路（每个页面）：
  ui-schema.md 页面名称 (P01: 订单列表)
    ↕ 一致？
  ui-schema.md 组件名 (OrderList ↔ 订单列表)
    ↕ 一致？
  ui-schema.md 路由 (/order/list ↔ 订单列表)
    ↕ 一致？
  ui-schema.md 侧边栏文案 (订单列表)
    ↕ 一致？
  user-stories.md 中的页面引用
    ↕ 一致？
  requirements.md 中的功能名称
```

重点检查：
- **侧边栏命名逻辑**：子菜单名称必须与父菜单形成语义层级（"订单管理 → 订单列表" ✅，"订单管理 → 充装记录" ❌）
- **组件名 = 页面名**：React 组件名必须与页面中文名直接对应

### 步骤 5: 生成报告与执行修正

**报告格式：**

```
## 术语一致性扫描报告

### 扫描范围
- 模块：{{MODULE_NAME}}
- 文档数：{{N}} 份
- 扫描术语数：{{N}} 个

### 发现问题

#### 不一致项（需修正）
| # | 术语 | 位置 | 当前用词 | 应改为 | 依据 |
|---|------|------|---------|-------|------|
| 1 | ... | requirements.md L42 | 充装记录 | 订单列表 | ui-schema P01 定义 |

#### 新增术语（需注册）
| # | 中文术语 | 英文术语 | 代码命名 | 所属分类 |
|---|---------|---------|---------|---------|
| 1 | ... | ... | ... | §1.1/§1.2/§1.3/§1.4 |

#### 使用规则违反
| # | 位置 | 违反内容 | 应改为 | 规则引用 |
|---|------|---------|-------|---------|
| 1 | ... | 使用"新建" | "新增" | §1.1 使用规则 |

### 修正汇总
- 不一致项：{{N}} 处修正
- 新增术语：{{N}} 个注册
- 规则违反：{{N}} 处修正
```

**执行修正**：按报告内容逐项修改文档和代码，优先级为 ui-schema > user-stories > requirements > architecture > ux-design。

### 步骤 6: 更新 STANDARDS.md

如有新增术语，将其追加到 STANDARDS.md 对应子表中：
- 业务实体 → §1.1
- 角色 → §1.2
- 模块/菜单 → §1.3
- 状态/枚举 → §1.4

新增术语必须同时填写：中文术语、英文术语、代码命名、说明。

## Prompt 模板

```
你是需求分析 Agent，负责对模块文档进行术语一致性扫描。

## 任务
对模块【{{MODULE_NAME}}】执行术语一致性检查。

## 输入文件
- 术语基准：docs/STANDARDS.md §1
- 模块文档：docs/features/{{DOMAIN}}/{{MODULE}}/ 下的所有 .md 文件
- 前端代码（如已存在）：frontend/src/features/{{DOMAIN}}/{{MODULE}}/ 下的 types.ts, constants.ts

## 执行步骤
请严格按照 docs/skills/analysis/glossary-management.md 中定义的步骤 1-6 执行。

## 关键要求
1. 以 STANDARDS.md §1 为唯一权威术语来源
2. 同一 UI 元素在 5 份文档中必须使用完全相同的名称
3. 页面名称、组件名、路由、菜单文案必须相互对应
4. 新增术语必须同时提供中英文和代码命名
5. 遵守术语使用规则（泛称规则、动作用语规则）
6. 输出修正清单，逐项说明修改位置和内容

## 输出
- 术语一致性报告（打印到控制台）
- 修正后的文档文件
- 更新后的 STANDARDS.md（如有新增术语）
```
